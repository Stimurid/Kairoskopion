# Source acquisition funnel

**Status:** golden rubric. Operationalises the canon's three depth
modes (canon §6) into a pool → shortlist → deep flow, with explicit
mapping to the V1–V8 depth levels and to the source allowlist A–J.

**Inputs:**
- Canon: [docs/VENUE_FUNNEL_AND_PROFILE_PACKAGE_V1.md](../../docs/VENUE_FUNNEL_AND_PROFILE_PACKAGE_V1.md).
- Source layer rubric: [venue_source_layer_map.md](venue_source_layer_map.md).
- Mavrinsky cluster gold: [mavrinsky_venue_side_gold.md](mavrinsky_venue_side_gold.md).
- Existing code: `services/venue_evidence_stack.py` (depth V1–V8),
  `services/venue_pool_discovery.py`, `adapters/venue/*` (6 adapters).

This file says **what is collected at which stage, with which adapter
calls, at what authority level, and what is deliberately deferred**.

---

## 0. Three stages

| stage | candidate count (typical) | depth levels (V) | active source categories (A–J) | typical cost |
|---|---|---|---|---|
| **Pool** | 30–80 | V1 (identity) + V2 (scope) | C + B + G | low — bulk metadata-API calls, no PDF, no LLM |
| **Shortlist** | 8–12 | + V3 (formal requirements) + V4 (5–10 article sample) | A + B + small D + G | medium — site fetch + light corpus |
| **Deep** | 3–5 | + V5 (editorial board seed) + V8 (35–80 articles, citation graph) | + E + F + H (selective full-text) | high — board scrape + heavy corpus |

These stages are not gates. A candidate stays in the registry at the
depth it reached; if later we need to deepen it, we escalate without
re-discovery.

---

## 1. Pool stage — 30–80 candidates, V1+V2 only

### What is collected

For each candidate:

| field | source category | example adapter |
|---|---|---|
| `canonical_name`, `aliases` | C — registry | OpenAlex Sources, Crossref, ISSN |
| `ISSN` / `eISSN` | C — registry | ISSN portal, OpenAlex Sources |
| `publisher` | C — registry | Crossref `member`, OpenAlex `host_organization` |
| `country`, `languages` | C — registry | OpenAlex Sources, DOAJ |
| `website_url`, `submission_url` | C + B | OpenAlex `homepage_url`, publisher portfolio page |
| `OpenAlex topics / Crossref subjects / DOAJ classification` | G — graph | OpenAlex `x_concepts`, DOAJ `subject` |
| `scope_summary` (vendor claim) | A — site | SnapshotCrawler (lightweight HTML capture, not deep parse) |
| `evidence_status` per field | autogenerated | `registry_card` for C, `vendor_claim` for A, `metadata_api` for G |

### What is NOT collected

- No editorial board scraping.
- No corpus articles fetched.
- No full text.
- No policy parsing beyond what the registry returns.
- No special-issue listing.

### Pool composition

For a typical Mavrinsky-like input, the pool is composed via canon
`venue_pool_discovery.query_planner`:

- 5–6 search queries per FPM discipline dimension above weight 0.15
  (continental_philosophy, philosophy_of_technology, media_philosophy,
  interface_theory, STS, etc.).
- Each query fans out across C and G adapters → ~10–20 candidates per
  cluster.
- Dedup via `venue_candidate_identity` (ISSN-first, name-normalised
  fallback).

### Scenario constraint enforcement

The pool respects `SubmissionScenario`:

- International scenario → exclude candidates whose `languages` is
  Russian-only and whose `indexing` is RU-only.
- Russian scenario → include the ВАК-perechen / РИНЦ adapter outputs
  (deferred: those adapters do not exist yet; for now, results are
  `UNKNOWN_NOT_VERIFIED` for that cluster).

### Expected access_status distribution

| status | typical share |
|---|---|
| `registry_card` (C hit) | 60–80 % of candidates have identity from registry |
| `metadata_api` (G hit) | 50–70 % have OpenAlex / Crossref enrichment |
| `vendor_claim` (A snapshot) | 30–50 % have scope text captured |
| `INACCESSIBLE` | 5–15 % (site 403, JS-only, dead URL) |
| `UNKNOWN_NOT_VERIFIED` | 5–10 % (ISSN exists but no enrichment) |

A run that produces > 30 % `INACCESSIBLE` at pool stage indicates a
provider-level problem (rate limit, blocked IP). Pool stage with
≥ 40 candidates and registry_card ≥ 50 % is healthy.

## 2. Shortlist stage — 8–12 candidates, +V3 +V4

After pool, candidates that survive a light scenario screen and a
light envelope check are promoted to shortlist. Promotion is **not**
automatic ranking — it is a methodological cut that names which
pathways (cluster 1–5) each candidate covers.

### What is added at shortlist depth

| field | source category | adapter |
|---|---|---|
| `aims_scope` (full text) | A | SnapshotCrawler with parse |
| `author_guidelines` | A | SnapshotCrawler, may need second fetch |
| `article_types declared` | A + B | guidelines + publisher template |
| `formal_submission_profile` (word_count, abstract, references, language, peer_review_model, OA/APC declared) | A + B | guidelines + publisher policy page |
| `light corpus sample` (5–10 latest articles, metadata only) | G + small D | OpenAlex `works` query by source_id |
| `pathway_decision` (which Mavrinsky cluster this candidate covers) | derived | scorer mapping § |

### What is NOT yet collected at shortlist

- Editorial board (V5) — deferred to deep stage unless candidate is
  flagged for cluster 5 (Russian philosophy), where board determines
  fit and must be sampled at shortlist already (canon §3.E + Mavrinsky
  gold §3).
- Citation graph (V8) — deferred.
- Special issues / CFP — deferred unless candidate has an active SI
  matching the article's pathway.
- Full text of corpus articles — deferred; metadata only.

### Pathway mapping

Each shortlisted candidate is labelled with `pathway_decision`
pointing to one of the 5 article-side pathways:

- `continental_media_philosophy`
- `philosophy_of_technology`
- `media_interface_theory`
- `STS_platform_studies`
- `HCI_design_theory`
- `RU_philosophy_regime` (cluster 5)

A candidate may carry multiple pathway decisions when its envelope
spans clusters (e.g., *Philosophy & Technology* covers both
cluster 1 boundary and cluster 2).

---

## 3. Deep stage — 3–5 candidates, +V5 +V8

Only candidates the operator selects for deep work get this. The
selection is human, not automatic.

### What is added at deep depth

| field | source category | adapter status |
|---|---|---|
| `EditorialBoardCloud` — full sample of 6–25 editors | E + G | **live adapter deferred** (see §6 below); current baseline accepts a fixture or operator-curated input |
| `PublishedCorpusHull` — 35–80 articles | D + G + H selective | existing `corpus_sampler` + `corpus_analyzer` + new `corpus_hull_builder` |
| `CitationExpectationProfile` — references aggregation over corpus + OpenCitations cited-by | G | OpenCitations adapter (exists, fixture mode) |
| `geographic_affinity` envelope | E + F | derived from editor and author affiliations |
| `temporal_position` | D + G | corpus reference-year distribution |
| `trust_compliance_profile` | C + A | DOAJ + COPE + OASPA cross-check |
| `section_models` / `issue_or_special_issue_models` (when relevant) | A + I | journal site + society channels |
| `tacit_venue_signals` (when operator provides) | J | operator-supplied notes with source-and-date stamp |

### What stays deferred even at deep

- Full text of every corpus article. Only enough full text to extract
  references and method markers.
- Live editorial board scraping for unstructured pages.
- Russian-language adapters (ВАК / РИНЦ / КиберЛенинка) — deferred to
  a separate sprint, see §5.

## 4. V1–V8 depth mapping

The canon's V1–V8 depth levels (see `services/venue_evidence_stack.py`)
map to the stages:

| V | name | stage | what triggers |
|---|---|---|---|
| V1 | identity | pool | always |
| V2 | scope claim | pool | always |
| V3 | formal requirements | shortlist | candidate survives scenario screen |
| V4 | light corpus (5–10) | shortlist | always at shortlist |
| V5 | editorial board seed | shortlist for cluster 5; deep for others | per cluster (canon §3.E) |
| V6 | full guidelines + policies | deep | always at deep |
| V7 | corpus standard (20–35) | deep | always |
| V8 | corpus deep (35–80) + citation graph + board cloud | deep | always |

`venue_evidence_stack.py` already implements the V1–V8 escalator; the
funnel here is a higher-level wrapper that names which stage triggers
which depth.

## 5. Russian scenario specifics (ВАК / РИНЦ / КиберЛенинка)

The Russian-philosophy cluster (cluster 5 in Mavrinsky venue-side gold)
has no working adapter yet. The current baseline behaviour:

- Under the international scenario, cluster 5 candidates are excluded
  by language filter.
- Under the Russian scenario, the funnel produces `UNKNOWN_NOT_VERIFIED`
  for the cluster-5 pool because:
  - No ВАК-perechen adapter exists. Candidates from this list cannot
    be enriched beyond their ISSN.
  - No РИНЦ / eLIBRARY adapter exists. Indexing claims for these
    venues are `vendor_claim`.
  - КиберЛенинка PDFs may be available for D corpus mining, but
    metadata about the venue itself must come from another source.

These adapters are backlog (canon Appendix C). They are intentionally
not in this baseline because:

1. ВАК / РИНЦ require careful handling of access patterns and rate
   limits.
2. The Mavrinsky international scenario does not require them.
3. Building them speculatively would violate the canon's "no
   speculative coverage" principle.

When the Russian scenario becomes the primary target, this baseline
will be the regression rubric the adapters land against.

## 6. Full-text resolvers (H) — corpus material only

Sources in category H — OA repositories, DOAJ full-text, Academia,
ResearchGate, Sci-Hub-like mirrors, personal library, user-uploaded
ZIPs — feed **D corpus mining** only.

| resolver | use | NOT used as |
|---|---|---|
| journal PDF / HTML | extract references, methods, language register | identity / policy / indexing |
| publisher OA copy | same | same |
| DOAJ full-text | same | same |
| OA institutional repository | same | same |
| author personal page | same | same |
| Academia.edu / ResearchGate | same, with reduced confidence | nothing else |
| Internet Archive Scholar | same | nothing else |
| Sci-Hub-like resolver | same, with risk flag | **never** for metadata |
| user-uploaded ZIP | same, with `selection_strategy: user_uploaded_pack` and `bias_notes` | nothing else |

The `FullTextResolver` adapter chain is **deferred**. For the current
baseline, the harness accepts a synthetic corpus and operates on
abstracts + metadata via existing OpenAlex/Crossref adapters.

## 7. EditorialBoardCloud — why live adapter is deferred

`EditorialBoardCloud` is the single most informative source for
cluster 5 (Russian philosophy) and a critical corroborator for
clusters 1–3. The adapter is deferred from this baseline because:

1. **Editorial board pages are heterogeneous.** Some are static HTML
   tables, some are JS-rendered React, some are PDFs of the masthead.
   No single scraping strategy works.
2. **Name → ORCID resolution is lossy.** Many editors have no ORCID,
   no clean affiliation string, or multiple ambiguous matches in
   OpenAlex Authors.
3. **Publication-history sampling is expensive.** Per editor, 5–10
   OpenAlex / Crossref calls. For a board of 20, that is 100–200 calls.
4. **The cloud is `INFERENCE` regardless.** Derived signals carry
   confidence flags; doing this in code without operator review risks
   AP3 (board as psychology).

The contract for the future live adapter is defined by canon §3.E
and source layer map §2 and §3.4. This baseline accepts:

- A synthetic / fixture `EditorialBoardCloud` for testing.
- An operator-curated `EditorialBoardCloud` as a JSON file under
  `private_inputs/`.
- A placeholder `EditorialBoardCloud` flagged `UNKNOWN_NOT_VERIFIED`
  for any venue where neither is supplied.

The adapter sprint is the next venue-side work after this baseline is
reviewed.

---

## 8. What this funnel deliberately defers

- Live editorial board adapter (E with HTTP scraping).
- ВАК / РИНЦ / КиберЛенинка adapters.
- Sherpa/RoMEO adapter integration (for self-archiving policies in
  `trust_compliance_profile`).
- Sci-Hub-like full-text resolvers.
- Telegram CFP channel adapter.
- Society newsletter / association site adapters.
- Live Google Scholar (high block risk; replaced by Semantic Scholar
  where the same data is needed).
- Closed-paid sources (Scopus / WoS / JCR) — these degrade to
  `UNKNOWN_NOT_VERIFIED` per canon §3.C.

Each is its own backlog item with its own contract.

---

## 9. Acceptance criteria

The funnel is accepted when, on a Mavrinsky-like input:

**A. Pool stage** produces ≥ 30 candidates with median
`access_status` ≥ `registry_card` and `INACCESSIBLE` rate < 20 %.
**B. Shortlist stage** produces 8–12 candidates with explicit
`pathway_decision` covering at least 3 of the 5 Mavrinsky clusters.
**C. Deep stage** produces 3 deep dossiers whose `VenueFieldPosition`
envelope overlaps the cluster gold per `mavrinsky_venue_side_gold.md`
§2.
**D. Cache discipline.** A repeat run within the freshness window
(canon §7 `fresh_sufficient`) makes zero new HTTP calls.

PASS: A + B + C + D. PARTIAL: A + B + (C OR D). FAIL: A or B FAIL.

End of funnel doc.
